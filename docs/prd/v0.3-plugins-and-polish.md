# v0.3 — Plugins, i18n & Settings hub (In progress)

Goal: make LocalBro *extensible* and *accessible*. Users pick their
language, pick their columns; developers ship behaviour as Packs.

## Features

### Internationalisation
- `react-i18next` + browser/OS language detection.
- Default locale resolves from `navigator.language`, falls back to
  `en`. `zh-CN` and `en` dictionaries ship with the app.
- Persisted override at `settings.get("ui.locale")`; changing it in
  Settings re-renders immediately.
- All user-visible strings in the core UI route through `t("…")`.

### Settings hub
- New ⚙ button at the Toolbar's far right.
- Modal with tabs:
  - **General** — language, default view mode.
  - **AI** — endpoint, key, model, policy (reuses `AiSettings`).
  - **Skins** — the existing picker embedded as a pane.
  - **About** — version, links to AI.md / PACKS.md, licence.
- Sidebar footer quick-buttons (🤖 / 🎨) are kept as shortcuts.

### Toolbar extras
- `🤖 AI` button right after `Hidden`, dispatches the `lb:open-ai`
  event that AI Panel already listens for.

### List column manager
- Applies to the default List view (Details view already picks up
  from it).
- Header row with user-toggleable columns: Name (pinned) / Size /
  Modified / Kind / Created / Extension.
- Column widths drag-to-resize via a handle between headers.
- Choice + widths persisted at `settings.set("ui.list.columns", …)`.
- First-run defaults match v0.2 (Name / Size / Modified).

### Plugin runtime (Task 9 MVP)
- Pack with `type: "plugin"` and `plugin.entry: <path>.js` is
  loaded via Vite/Tauri's `convertFileSrc` → dynamic `import()`.
- Loaded module's `default` export is `async ({ api, register }) =>
  void`; `register` lets it contribute:
  - `previewAdapter({ id, priority, match, component })`
  - `aiTool({ name, kind, description, parameters, pathFields, run })`
- The `api` surface handed to the plugin is a whitelist of read-only
  wrappers over `src/api.ts`, filtered by the `permissions` declared
  in the manifest. Anything not declared returns a rejected Promise.
- Enabled state stored at `settings.get("plugins.enabled")` — users
  flip plugins in the Settings > Skins pane (renamed "Packs" later).

## Non-goals (v0.3)
- Native sidecar processes (v0.4).
- Per-plugin sandbox iframes or web workers (relying on Tauri's
  WebView isolation for now).
- Auto-update of installed Packs.
- Locales beyond `zh-CN` and `en`.

## Design notes
- **Why no Worker / iframe sandbox yet?** — Every plugin already runs
  inside Tauri's single renderer, same origin as the app. A Worker
  wouldn't buy security (shared window.crypto, shared FS via `api.*`).
  Real security would require moving `api.*` into an IPC-bridged
  proxy, which is a v0.4 conversation once real third-party plugins
  exist.
- **Why the same `api` as the host?** — Plugins shouldn't have to
  learn a second API. The permissions table in `PACKS.md` is the
  contract; missing permissions surface as `PermissionDenied`.
- **Why keep Sidebar shortcuts after adding Settings hub?** — Power
  users will use the AI button 20× a day. Burying it in a modal would
  regress v0.2 UX.

## Decision log
- *2026-05* — react-i18next over lighter custom solution: ICU plural
  rules and lazy-load namespaces are worth the 35KB gzip for a
  product targeting non-English users.
- *2026-05* — Column config stored per-user, not per-directory. Scope
  creep ("remember columns in `~/Pictures`") deferred.
- *2026-05* — Plugin runtime loads over `tauri://convertFileSrc`. The
  `blob:` alternative cannot resolve relative imports.
